# DynASM — Comprehensive Documentation

## Overview

**DynASM** (*Dynamic Assembler*) is a pre-processing assembler and encoding engine for code generation runtimes. Originally designed and implemented for LuaJIT by Mike Pall, it allows you to write assembly code inline with C code, and have it assembled at runtime into executable machine code.

- **Version**: 1.5.0
- **Author**: Mike Pall
- **License**: MIT
- **URL**: [https://luajit.org/dynasm.html](https://luajit.org/dynasm.html)

The source lives under `.vendor/LuaJIT/dynasm/` and consists of two halves:

1. **Pre-processor** (`dynasm.lua` + arch modules) — a Lua program that reads `.dasc` files (mixed C + assembly) and outputs pure C with embedded action lists.
2. **Encoding engine** (the `.h` files) — C code that consumes action lists at runtime and produces raw machine code bytes.

## Architecture

```
┌──────────────────────────────────────────────────────────┐
│  .dasc input file (C + |assembly)                        │
│                                                          │
│  const void *code:                                       │
│  |.arch x86                                              │
│  |.code                                                  │
│  | mov eax, 42                                           │
│  | ret                                                   │
└──────────────────────┬───────────────────────────────────┘
                       │ dynasm.lua (pre-processor)
                       ▼
┌──────────────────────────────────────────────────────────┐
│  .c / .h output (pure C + static action list array)      │
│                                                          │
│  static const unsigned char actions[21] = { ... };        │
│  ...                                                      │
│  dasm_put(Dst, 0, ...);   // calls into C encoding engine │
└──────────────────────┬───────────────────────────────────┘
                       │ compile with C compiler
                       │ link against encoding engine (.h)
                       ▼
┌──────────────────────────────────────────────────────────┐
│  Runtime: action list ──► dasm_put() ──► dasm_link()     │
│                        ──► dasm_encode() ──► raw bytes    │
└──────────────────────────────────────────────────────────┘
```

## Using DynASM from LuaJIT

DynASM is **not** part of the installed LuaJIT runtime — it lives only in the
LuaJIT source tree under `dynasm/`. A LuaJIT user has three practical ways to
use it, depending on the use case.

---

### Approach 1: Build-time pre-processing (recommended for most users)

This is the standard DynASM workflow. You write assembly inline with C in a
`.dasc` file, pre-process it into pure C, compile to a shared library, and call
it from LuaJIT via the FFI.

**Step 1 — Write your `.dasc` file** (`myjit.dasc`):
```c
|.arch x64
|.section code
|.actionlist actions

// Function: int add(int a, int b)
|->add:
| lea eax, [ecx+edx]    // Windows x64: a=ecx, b=edx
| ret
```

**Step 2 — Pre-process with dynasm.lua:**
```bash
luajit /path/to/LuaJIT/dynasm/dynasm.lua -o myjit.h myjit.dasc
```

This produces a C header `myjit.h` containing a `static const unsigned char
actions[]` array and a function that calls `dasm_put()` to encode the
instructions.

**Step 3 — Write a small C driver** (`myjit.c`):
```c
#define DASM_CHECKS 1
#include "dasm_proto.h"
#include "dasm_x86.h"
#include "myjit.h"

// Generated by dynasm — calls dasm_put to encode the add function
static void build_add(dasm_State **Dst) {
  dasm_growpc(Dst, 10);
  dasm_setup(Dst, actions);
  // ... generated dasm_put() calls from myjit.dasc ...
  size_t sz;
  dasm_link(Dst, &sz);
  void *buf = malloc(sz);
  dasm_encode(Dst, buf);
  return buf;
}
```

**Step 4 — Compile to a shared library:**
```bash
gcc -shared -fPIC -I/path/to/LuaJIT/dynasm -I/path/to/LuaJIT/src \
    -o myjit.so myjit.c
```

**Step 5 — Call from LuaJIT via FFI:**
```lua
local ffi = require("ffi")

ffi.cdef[[
  typedef struct dasm_State dasm_State;
  void *build_add(dasm_State **Dst);
]]

local myjit = ffi.load("./myjit")
local Dst = ffi.new("dasm_State *[1]")
local buf = myjit.build_add(Dst)

local add = ffi.cast("int (*)(int, int)", buf)
print(add(3, 4))  --> 7
```

This is how most projects use DynASM — the assembly is fixed at build time,
and only the encoding (label resolution, address patching) happens at runtime.

---

### Approach 2: Runtime code generation from Lua

If you want to **dynamically** generate assembly from within a running LuaJIT
script, you can load the DynASM Lua modules. They are plain Lua and run fine
under LuaJIT. However, you still need the **C encoding engine** — compile it as
a shared library and bind it with FFI.

**Step 1 — Build the encoding engine as a shared library** (`dasmlib.c`):
```c
#define DASM_CHECKS 1
#include "dasm_proto.h"
#include "dasm_x86.h"   // pick your arch

// Provide the required extern/memory hooks
#define DASM_EXTERN(ctx, addr, idx, type)  0
// Memory allocator: use realloc/free or custom allocator
```

Compile:
```bash
gcc -shared -fPIC -I/path/to/LuaJIT/dynasm \
    -I/path/to/LuaJIT/src -o dasmlib.so dasmlib.c
```

**Step 2 — Load modules from LuaJIT:**
```lua
local ffi = require("ffi")

ffi.cdef[[
  typedef struct dasm_State dasm_State;
  void dasm_init(dasm_State **Dst, int maxsection);
  void dasm_free(dasm_State **Dst);
  void dasm_setupglobal(dasm_State **Dst, void **gl, unsigned int maxgl);
  void dasm_growpc(dasm_State **Dst, unsigned int maxpc);
  void dasm_setup(dasm_State **Dst, const void *actionlist);
  void dasm_put(dasm_State **Dst, int start, ...);
  int  dasm_link(dasm_State **Dst, size_t *szp);
  int  dasm_encode(dasm_State **Dst, void *buffer);
]]

local C = ffi.load("./dasmlib")

-- Add dynasm/ to Lua path
package.path = "/path/to/LuaJIT/dynasm/?.lua;" .. package.path

-- Now you can load architecture modules
local dasm_x86 = require("dasm_x86")
-- dasm_x86 exposes: setup(), passcb(), mergemaps(), section(),
--                   dumparch(), dumpdef()
```

In practice, driving DynASM programmatically from Lua is complex — the modules
were designed for use by the `dynasm.lua` CLI, not as a clean API. Most users
who need runtime code generation use Approach 3 instead.

---

### Approach 3: Pre-encode templates, patch at runtime

This is how LuaJIT's own JIT compiler works, and it's the most powerful
pattern. You write **parameterized** assembly templates in a `.dasc` file,
pre-process them once, then at runtime you call `dasm_put()` with
**runtime-determined values** for registers, labels, and immediates.

**Key enabling features:**

- **VREG (variable registers)**: Register numbers are determined at runtime.
  The pre-processor emits actions that the C encoder fills in later.
  ```asm
  | mov RAd, [RB + IR:op1]    // RA, RB bound at runtime
  ```

- **PC Labels**: Branch targets determined at runtime:
  ```asm
  | jmp =>exit_label
  ```
  Then at runtime: `dasm_put(Dst, offset, pc_of_exit_label)`

- **Dynamic immediates**: Values provided as arguments to `dasm_put()`:
  ```asm
  | mov eax, $imm
  ```

**Complete example** (`template.dasc`):
```c
|.arch x64
|.section code
|.actionlist actions

// Template: add immediate to register, jump to exit
|->add_imm:
| add RAd, $imm
| jmp =>exit
```

At runtime, the LuaJIT JIT compiler repeatedly calls:
```c
dasm_put(Dst, offset,
  REG_RA,     // bind RA to actual register
  42,         // immediate value
  exit_label  // PC label for exit
);
```

Each call produces different machine code from the same template.

---

### Summary: Which approach to use?

| Use case | Approach |
|---|---|
| I want to write fixed assembly, compile it, call it from Lua | **Approach 1**: build-time `.dasc` |
| I want to generate varying machine code at runtime based on Lua logic | **Approach 3**: pre-encode templates, patch at runtime via `dasm_put()` |
| I'm building a JIT compiler in Lua | **Approach 3** — this is exactly what LuaJIT does internally |
| I just want to run raw bytes as machine code | Use `ffi.cast` on a byte buffer — no DynASM needed |

## How It Works — End to End

### 1. The Pre-processor (dynasm.lua)

`dynasm.lua` reads a `.dasc` file that contains a mix of plain C and assembler lines. Assembler lines are indicated by a leading `|` (pipe) character.

**Example `.dasc` file:**
```c
|.arch x86
|.section code, data
|.actionlist actions

static void generate(dasm_State **Dst) {
  dasm_growpc(Dst, 10);
  dasm_setup(Dst, actions);
  |.code
  | mov eax, 42
  | ret
  dasm_link(Dst, &sz);
  dasm_encode(Dst, buf);
}
```

#### Processing Pipeline

1. **Parse options** — The CLI parses short/long options (see below).
2. **Load architecture module** — the `.arch` pseudo-op loads an arch-specific Lua module (e.g., `dasm_x86.lua`) which merges its opcode map and define map with the core maps.
3. **Process line by line** — each line is either:
   - **Plain C**: emitted directly to the output buffer.
   - **Assembler line** (leading `|`): the pre-processor splits it into statements (separated by `;`), expands `#define` substitutions, splits each statement into opcode + parameters, looks up the handler, and invokes it.
   - **Double-pipe** (`||`): C code emitted as-is, even from inside macros (avoids echoing and parsing).
4. **Action list accumulation** — as opcodes are processed, they write "actions" into an in-memory buffer. Periodically the buffer is flushed as calls to `dasm_put(Dst, offset, args...)` in the output C file.
5. **Duplicate elimination** — the x86 module deduplicates action list chunks to keep the final array compact.
6. **Output** — writes a header comment, all buffered C/action-list lines, and optionally dumps defines/macros/sections.

### 2. The Action System

Actions are the intermediate representation between the Lua pre-processor and the C encoding engine. They form a bytecode for machine code generation.

**For x86**, the action list is a `const unsigned char` array. Each encoded instruction becomes a sequence of action bytes:
- **Raw opcode bytes** (0x00–0xE8) placed directly into the code buffer.
- **Action codes** (0xE9–0xFF) that trigger special processing at encode time (e.g., resolve labels, insert immediates, compute ModRM/SIB).

Action codes for x86:
```
DISP      (0xE9) — defer displacement to encode time
IMM_S     (0xEA) — signed byte immediate
IMM_B     (0xEB) — unsigned byte immediate
IMM_W     (0xEC) — unsigned word immediate
IMM_D     (0xED) — dword immediate
IMM_WB    (0xEE) — word immediate, buffer position
IMM_DB    (0xEF) — dword immediate, buffer position
VREG      (0xF0) — variable register
SPACE     (0xF1) — reserve space
SETLABEL  (0xF2) — assign label address
REL_A     (0xF3) — absolute relocation
REL_LG    (0xF4) — relative to local/global label
REL_PC    (0xF5) — relative to PC label
IMM_LG    (0xF6) — immediate from local/global label
IMM_PC    (0xF7) — immediate from PC label
LABEL_LG  (0xF8) — define local/global label
LABEL_PC  (0xF9) — define PC label
ALIGN     (0xFA) — alignment
EXTERN    (0xFB) — external symbol
ESC       (0xFC) — escape byte
MARK      (0xFD) — marker for REX/VEX optimization
SECTION   (0xFE) — section switch (terminal)
STOP      (0xFF) — end of action list (terminal)
```

For ARM/ARM64/MIPS/PPC, the action list uses 32-bit words (`const unsigned int`) where the upper 16 bits are the action code and the lower 16 bits carry immediate data.

### 3. The Encoding Engine (C headers)

The C encoding engine works in **three passes**:

#### Pass 1 — `dasm_put()`
- Walks the action list, processes simple actions immediately, and records the offset estimates.
- For labels and relocations: chains them together. A label stores `-pos` (its position). A forward reference chains through `*pl = pos`.
- Outputs an estimated offset for each label location (pass 1 estimate).

#### Pass 2 — `dasm_link()`
- Walks all sections, resolves label offsets using the final offsets computed in pass 1.
- Applies alignment (shrinks the offset by the align mask).
- Calculates the total code size.
- Collapses undefined global label chains into negative markers (`-idx`).

#### Pass 3 — `dasm_encode()`
- Walks the action list again, this time writing actual machine code bytes into the output buffer.
- Resolves labels using final positions.
- Encodes immediates, computes ModRM/SIB/REX/VEX for x86.
- Handles external symbol resolution via the `DASM_EXTERN` callback.
- Validates range constraints (branch displacement, immediate sizes).

## Core API (C Side)

All defined in `dasm_proto.h` and the arch-specific `.h` files.

### Data Types

```c
typedef struct dasm_State dasm_State;   // Opaque encoder state

// Common macros for declaring state pointer:
// Dst_DECL defaults to: dasm_State **Dst
// Dst_REF  defaults to: (*Dst)
```

### Lifecycle Functions

```c
// Initialize state with maxsection code sections.
void dasm_init(Dst_DECL, int maxsection);

// Free all allocated memory.
void dasm_free(Dst_DECL);
```

### Setup Functions

```c
// Setup global label array. Must be called BEFORE dasm_setup().
// gl: array of void* pointers for global labels.
// maxgl: max number of global labels.
void dasm_setupglobal(Dst_DECL, void **gl, unsigned int maxgl);

// Grow PC label array. Can be called before or after dasm_setup().
void dasm_growpc(Dst_DECL, unsigned int maxpc);

// Setup encoder with the action list produced by the pre-processor.
void dasm_setup(Dst_DECL, const void *actionlist);
```

### Encoding Functions

```c
// Feed encoder with actions. start is the offset into the action list.
// Additional arguments depend on the action list contents (labels, immediates).
// Generated by the pre-processor; not typically called manually.
void dasm_put(Dst_DECL, int start, ...);

// Link sections after all dasm_put() calls. Returns total code size in *szp.
// Returns DASM_S_OK on success, or an error status code.
int dasm_link(Dst_DECL, size_t *szp);

// Encode all sections into the provided buffer.
// Buffer must be at least the size returned by dasm_link().
int dasm_encode(Dst_DECL, void *buffer);
```

### Utility Functions

```c
// Get the offset of a PC label. Returns -1 if undefined, -2 if unused.
int dasm_getpclabel(Dst_DECL, unsigned int pc);

// Optional sanity checker for step-by-step encoding.
// Returns DASM_S_OK or an error status.
int dasm_checkstep(Dst_DECL, int secmatch);
```

### Status Codes

```c
DASM_S_OK          (0x00000000)  // Success
DASM_S_NOMEM       (0x01000000)  // Out of memory
DASM_S_PHASE       (0x02000000)  // Phase error (size mismatch between passes)
DASM_S_MATCH_SEC   (0x03000000)  // Section mismatch
DASM_S_RANGE_I     (0x11000000)  // Immediate out of range
DASM_S_RANGE_SEC   (0x12000000)  // Section index out of range
DASM_S_RANGE_LG    (0x13000000)  // Local/global label out of range
DASM_S_RANGE_PC    (0x14000000)  // PC label out of range
DASM_S_RANGE_VREG  (0x15000000)  // Variable register out of range (x86/ARM64)
DASM_S_RANGE_REL   (0x15000000)  // Relocation out of range (ARM/MIPS/PPC)
DASM_S_UNDEF_L     (0x21000000)  // Undefined label (x86)
DASM_S_UNDEF_LG    (0x21000000)  // Undefined local/global label
DASM_S_UNDEF_PC    (0x22000000)  // Undefined PC label
```

### Customizable Macros

```c
// Override memory growth: (context, type, pointer, size_var, needed)
#ifndef DASM_M_GROW
#define DASM_M_GROW(ctx, t, p, sz, need) \
  do { size_t _sz = (sz), _need = (need); \
    if (_sz < _need) { \
      if (_sz < 16) _sz = 16; \
      while (_sz < _need) _sz += _sz; \
      (p) = (t *)realloc((p), _sz); \
      if ((p) == NULL) exit(1); \
      (sz) = _sz; \
    } \
  } while(0)
#endif

// Override memory free: (context, pointer, size)
#ifndef DASM_M_FREE
#define DASM_M_FREE(ctx, p, sz)  free(p)
#endif

// Override state pointer declaration
#ifndef Dst_DECL
#define Dst_DECL  dasm_State **Dst
#endif

// Override state pointer dereference
#ifndef Dst_REF
#define Dst_REF   (*Dst)
#endif

// Override storage class
#ifndef DASM_FDEF
#define DASM_FDEF  extern
#endif
```

## DynASM Language Reference

### File Format

A `.dasc` file contains a mix of C code and assembler directives. Assembler lines begin with `|` (pipe) at any indentation level.

```c
|.arch x86        // First statement must declare architecture

void my_codegen(dasm_State **Dst) {
  |.code
  | mov eax, 42
  | ret
}
```

### Architecture Declaration

The first statement must be `.arch <name>`:
```
.arch x86
.arch x64        (same as x86 but in 64-bit mode)
.arch arm
.arch arm64
.arch mips
.arch mips64
.arch ppc
```

### Sections

Code sections let you organize generated code into separate buffers:

```
|.section code, data
```

Generates:
```c
#define DASM_SECTION_CODE  0
#define DASM_SECTION_DATA  1
#define DASM_MAXSECTION    2
```

Switch sections with `.code`, `.data`, etc.:
```
|.code
| mov eax, 1
|.data
|.byte 0x00, 0x01, 0x02
```

### Labels

Three kinds of labels are supported:

| Syntax | Type | Range |
|---|---|---|
| `1:` through `9:` | Local, numeric | Forward/backward, nestable |
| `->name:` | Global label | Cross-module (via `void*` array) |
| `=>expr:` | PC label | Programmatic access via `dasm_getpclabel()` |

Local label references:
```
| jmp >1       // forward reference to label 1
|1:
| jmp <1       // backward reference to label 1
```

Global label references:
```
|->mylabel:
| jmp ->mylabel
```

PC label references:
```
|=>my_pc:
| jmp =>my_pc
```

Extern label references (for linking against external symbols):
```
| call extern my_func
```

Label with address assignment (x86 only):
```
|->label, &my_c_variable
```

### Defines (Substitution)

```asm
|.define FOO, 42
| mov eax, FOO         // expands to: mov eax, 42

|.define BAR           // BAR = 1 (default)
```

Command-line defines:
```
dynasm -D NAME=value
dynasm -U NAME          # undefine
```

### Conditional Assembly

```
|.if X64
|  mov rax, rcx
|.elif ARM64
|  mov x0, x1
|.else
|  mov eax, ecx
|.endif
```

Condition expressions are evaluated as Lua expressions with no globals (all unknown identifiers evaluate to `nil`/`false`). Numeric values evaluate as truthy (except `0` which is false).

### Macros

Define reusable sequences of assembly:

```asm
|.macro push_two, a, b
| push a
| push b
|.endmacro

| push_two eax, ebx     // expands to push eax; push ebx
```

Macro parameters are substituted lexically. Token-pasting is done with `..`:
```asm
|.macro load_rm, sz
| mov r..sz, [rax]
|.endmacro
```

Macros can be nested and use conditional assembly internally. Recursive macro invocation is detected and produces a fatal error.

### Captures

Captures let you collect emitted code into named buffers and replay them later (useful for generating code once and duplicating it):

```asm
|.capture prologue
| push rbp
| mov rbp, rsp
|.endcapture

|.dumpcapture prologue   // emits push rbp; mov rbp, rsp
```

Every capture must have a corresponding `.dumpcapture`.

### Data Directives (x86)

```
|.byte   0x90, 0x90       // raw bytes
|.sbyte  -1               // signed byte
|.word   0x1234           // 16-bit word
|.dword  0x12345678       // 32-bit dword
|.qword  0x123456789ABCDEF0  // 64-bit qword
|.aword  some_label       // address-sized word
|.long   0xDEADBEEF       // alias for .dword
|.quad   0x1234...        // alias for .qword
|.addr   some_label       // address-sized
```

### Data Directives (ARM/ARM64)

```
|.long   0x12345678       // 32-bit word
|.quad   0x1234...        // 64-bit qword
|.addr   some_label       // pointer-sized
```

### Alignment and Spacing

```
|.align 4                 // align to 4-byte boundary (power of 2)
|.align 16                // align to 16 bytes
|.space 8                 // reserve 8 bytes (zero-filled)
|.space 8, 0x90           // reserve 8 bytes filled with 0x90 (NOP)
```

### Type Definitions

User-defined types for structured memory access (x86, ARM, ARM64):

```
|.type MYSTRUCT, MyStruct, rbx->    // register override optional
```

Generates a `DtX(_V)` macro for computing offsets:
```c
#define Dt1(_V) (int)(ptrdiff_t)&(((MyStruct *)0)_V)
```

Then in assembly:
```
| mov eax, [rdx + MYSTRUCT:field_name]
```

### Action List and Globals Emission

```
|.actionlist actions        // emit "static const unsigned char actions[] = {...};"
|.globals GLOBAL_           // emit "enum { GLOBAL_foo, GLOBAL_bar, GLOBAL__MAX };"
|.globalnames global_names  // emit "static const char *const global_names[] = {...};"
|.externnames extern_names  // emit "static const char *const extern_names[] = {...};"
```

## CLI Usage

```
dynasm [OPTION]... INFILE.dasc|-

Options:
  -h, --help           Display help
  -V, --version        Display version and copyright

  -o, --outfile FILE   Output file (default: stdout)
  -I, --include DIR    Add include search path

  -c, --ccomment       Use /* */ comments for assembler lines
  -C, --cppcomment     Use // comments for assembler lines (default)
  -N, --nocomment      Suppress assembler line comments in output
  -M, --maccomment     Show macro expansions as comments

  -L, --nolineno       Suppress #line directives in output
  -F, --flushline      Flush action list after every assembler line

  -D NAME[=SUBST]      Define a substitution
  -U NAME              Undefine a substitution

  -P, --dumpdef        Dump defines, macros, captures (repeat for verbose)
  -A, --dumparch ARCH  Dump architecture description and exit
```

## Architecture-Specific Details

### x86 / x64

**File**: `dasm_x86.lua` + `dasm_x86.h`

The most feature-complete module. Covers:
- Base x86/x64 integer instructions
- FPU (x87) instructions
- MMX, SSE, SSE2, SSE3, SSSE3, SSE4.1, SSE4.2
- AVX, AVX2, FMA3
- BMI1, BMI2, ADX
- AES-NI, PCLMULQDQ

**Register naming**:
- `rax`–`rdi` (qword, x64), `eax`–`edi` (dword), `ax`–`di` (word), `al`–`bh` (byte)
- `xmm0`–`xmm15` (oword SSE), `ymm0`–`ymm15` (yword AVX)
- `st0`–`st7` (x87 FP stack)

**Operand size keywords**: `byte`, `word`, `dword`, `qword`, `oword`, `yword`, `tword`

**Memory addressing**: `[base + index*scale + displacement]`
```asm
| mov eax, [ebx + esi*4 + 8]
| mov rax, [rip + mylabel]        // RIP-relative (x64)
| mov eax, [fs:rbx]               // segment override (x86)
```

**VEX encoding** is automatic when using AVX instructions:
```asm
| vaddps ymm0, ymm1, ymm2
```

**Variable registers** (for JIT compilers that don't know register numbers at pre-processing time):
```
|.type IR, struct IR, Rax()
| mov eax, [ebx + IR:op1]
```

### ARM64 (AArch64)

**File**: `dasm_arm64.lua` + `dasm_arm64.h`

Covers:
- Base integer instructions (add, sub, mul, div, bitwise ops)
- Load/store (ldr, str, ldp, stp, with all addressing modes)
- Branches (b, bl, blr, br, ret, cbz, cbnz, tbz, tbnz, conditional branches)
- Floating point (fadd, fsub, fmul, fdiv, fcvt, fmov, fcmp, etc.)
- Bitfield operations (sbfm, bfm, ubfm)
- Pointer authentication (PAC) for ARM64e
- Branch Target Identification (BTI)

**Register naming**: `x0`–`x30`, `w0`–`w30`, `sp`, `xzr`, `wzr`, `lr` (alias for `x30`)

**Addressing modes**:
```asm
| ldr x0, [x1]           // base register
| ldr x0, [x1, #8]       // base + immediate offset
| ldr x0, [x1, #8]!      // pre-indexed (write-back)
| ldr x0, [x1], #8       // post-indexed
| ldr x0, [x1, x2, lsl #3]  // base + scaled index
| ldr x0, [x1, x2, sxtx]    // base + extended index
```

**Immediates**: ARM64 immediates use bit-pattern encoding:
- `#imm12`: 12-bit unsigned or `0xXXX00000` form
- `#imm13`: 13-bit bitmask (replicated bit pattern)
- `#imm6`: 6-bit for shift amounts

### ARM32

**File**: `dasm_arm.lua` + `dasm_arm.h`

Covers ARMv5TE through ARMv7, including:
- Core integer instructions (with `s` suffix for flag-setting)
- Multiply-accumulate
- Load/store with all ARM addressing modes
- LDM/STM (push/pop)
- VFP floating point
- Selected ARMv6/v6T2 instructions (SIMD, saturating, bitfield)

**Register naming**: `r0`–`r15` (`sp`=`r13`, `lr`=`r14`, `pc`=`r15`)

**Condition codes** are appended to instruction mnemonics:
```asm
| addeq r0, r1, r2       // ADD if equal
| movne r0, #42           // MOV if not equal
```

**Shift operands** on data-processing instructions:
```asm
| add r0, r1, r2, lsl #3
| mov r0, r1, lsr r2       // register-specified shift
```

### MIPS32 / MIPS64

**File**: `dasm_mips.lua` + `dasm_mips.h`

Covers MIPS32 and MIPS64 instruction sets. Supports MIPSr6 via `-D MIPSR6=1`.

**Action codes** (simpler than x86): `STOP`, `SECTION`, `ESC`, `REL_EXT`, `ALIGN`, `REL_LG`, `LABEL_LG`, `REL_PC`, `LABEL_PC`, `IMM`, `IMMS`

### PowerPC / PPC64

**File**: `dasm_ppc.lua` + `dasm_ppc.h`

Covers PPC32 and PPC64. Extended by Caio Souza Oliveira.

**Action codes**: `STOP`, `SECTION`, `ESC`, `REL_EXT`, `ALIGN`, `REL_LG`, `LABEL_LG`, `REL_PC`, `LABEL_PC`, `IMM`, `IMMSH`

## Typical Integration Pattern

```c
#include "dasm_proto.h"
#include "dasm_x86.h"    // or dasm_arm64.h, etc.

// 1. Initialize
dasm_State *D = NULL;
void *globals[GLOBAL__MAX] = {0};

dasm_init(&D, DASM_MAXSECTION);
dasm_setupglobal(&D, globals, GLOBAL__MAX);
dasm_growpc(&D, MAX_PC_LABELS);

// 2. Setup (action list from generated file)
dasm_setup(&D, actions);

// 3. Generate (calls to dasm_put embedded in generated C)
// ... (code that uses dasm_put from generated file) ...

// 4. Link
size_t sz;
int rc = dasm_link(&D, &sz);
if (rc != DASM_S_OK) { /* error */ }

// 5. Encode
void *buf = malloc(sz);
rc = dasm_encode(&D, buf);
if (rc != DASM_S_OK) { /* error */ }

// 6. Use
// buf now contains executable machine code
// globals[i] now contains pointers to global labels

// 7. Cleanup (or reuse with dasm_setup)
dasm_free(&D);
```

### Step-by-Step Encoding (Incremental Generation)

Some JIT compilers generate code in discrete steps. Use `dasm_checkstep()` between steps:

```c
// After each chunk of dasm_put calls:
if (dasm_checkstep(&D, -1) != DASM_S_OK) { /* error */ }
```

## Error Handling Strategy

- The pre-processor emits errors to stderr with `filename:line: error: message`.
- Up to 20 errors are reported before messages are suppressed.
- Fatal errors (`.fatal`) stop processing immediately.
- Warnings (via `wwarn`) don't stop processing.
- The C encoding engine returns status codes from `dasm_put()`, `dasm_link()`, and `dasm_encode()`.
- When `DASM_CHECKS` is defined, range checks are performed during encoding. Without it, checks are elided for performance.

## Internal Design Notes

### Duplicate Action List Elimination (x86 only)

The x86 module concatenates action list bytes and searches for duplicate chunks using `string.find`. This keeps the final C array compact, which is important for I-cache efficiency.

### REX/VEX Shrinkage (x86 only)

The x86 pre-processor tracks how many variable register (VREG) actions contribute to REX/VEX prefix shrinkage and optimizes encoding depth accordingly.

### Two Encoding Styles

The architecture modules fall into two families:

1. **Byte-oriented** (x86): Action list is a `const unsigned char[]`. Opcode bytes are placed directly. Action codes have values 0xE9–0xFF. Complex encoding due to variable-length instructions, ModRM/SIB/REX/VEX prefixes.

2. **Word-oriented** (ARM, ARM64, MIPS, PPC): Action list is a `const unsigned int[]`. Each entry is `(action_code << 16) | immediate`. Action codes start at 0. All instructions are fixed 32-bit words, so encoding is simpler — the template string directly encodes the base instruction word, and actions fill in register/immediate fields by OR-ing bits.

### Template Strings

Each opcode maps to a template string that encodes:
- **ARM/MIPS/PPC**: An 8-hex-digit base instruction + character sequence specifying how to parse operands and merge them into the instruction word.
- **x86**: A colon-separated pattern string that describes operand modes, sizes, opcode bytes, and how to emit ModRM/SIB/VEX.

The `.template__` handler iterates through `|`-separated alternatives in the template and attempts to match operand modes and sizes.
