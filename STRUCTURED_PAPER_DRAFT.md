# StructurED: Structural Decomposition for LLM Architecture Search

## Abstract

We present **StructurED**, a method for improving LLM reasoning on complex architecture and code-generation tasks through structural decomposition into specialized, isolated agent contexts. Inspired by PLANSEARCH (Wang et al., 2024), which demonstrated that searching over natural language plans outperforms direct code search, StructurED replaces PLANSEARCH's combinatorial observation sampling with **process-level agent isolation**. Each agent operates in a separate LLM process with its own system prompt, tool set, and context window. A shared append-only context artifact accumulates cumulative reasoning across agents, producing a naturally attention-optimal document shape at each step.

We find that this structural separation produces diversity in reasoning paths not through enumerative sampling but through genuine divergence of agent roles. The approach is KISS — a few agents, each with a crisp role, connected by a single growing context file — and maps directly to the transformer attention mechanism's known properties: U-shaped attention (top/bottom of context dominate) and token-continuation reasoning (each agent's context tells the "story" leading to its correct output).

## 1. Introduction

The bitter lesson (Sutton, 2019) posits that two forms of scaling dominate: learning and search. For LLMs, scaling inference compute through repeated sampling has shown diminishing returns (Brown et al., 2024; Chen et al., 2024) because models repeatedly produce highly similar, incorrect outputs. PLANSEARCH (Wang et al., 2024) demonstrated that the right axis for search is not tokens or programs but *natural language plans*. By generating combinatorial subsets of observations about a problem and deriving plans from each subset, PLANSEARCH achieves significantly higher diversity and pass rates than repeated sampling.

However, PLANSEARCH achieves this diversity through combinatorial enumeration within a *single LLM context*. This has two limitations:

1. **Observation space is combinatorial**. Generating all subsets of size ≤ 2 from a pool of observations grows quadratically. For code generation problems with small observation spaces this is tractable; for architecture problems with large codebases it is not.

2. **Single context, single model**. All observations, plans, and code are generated by the same model in the same context window. Diversity comes from prompt engineering and high-temperature sampling, not from structural differences in what the model sees or can do.

We propose **StructurED** (Structural Exploration through Decomposition), which replaces combinatorial observation sampling with **process-level agent isolation**. Each agent runs as a separate process with:
- A different system prompt tailored to a specific role
- A different set of tools (some agents can read and search, some can also spawn sub-agents, some are read-only)
- A fully isolated context window (no shared token space)
- An append-only shared context artifact (`workflow.md`) that accumulates cumulative reasoning

## 2. Method

### 2.1 Agent Pipeline

StructurED defines nine agents, each with a crisp role:

| Agent | Role | Tools |
|-------|------|-------|
| Scout | Gather maximum first-order facts: files, types, functions, dependencies, data flows | read, grep, find, ls, bash |
| Knowledge Builder | Produce second-order observations: patterns, tensions, edge cases, implicit constraints. Can spawn scouts. | read, grep, find, ls, subagent |
| Approach Proposer | Generate 3 genuinely distinct architectural approaches with different tradeoffs | read, grep, find, ls, subagent |
| Critic | Evaluate each approach on coupling, cohesion, migration cost, philosophy fit, risk | read, grep, find, ls, bash |
| Documenter | Faithfully capture a decision in a self-sustaining artifact. No new analysis. | read |
| Edit Planner | Produce file-by-file, line-level implementation plan | read, grep, find, ls, subagent |
| Worker | Execute implementation in isolated context | (full) |
| Reviewer | Code review via git diff/log/show | read, grep, find, ls, bash |
| Planner | Simpler code-level planning | read, grep, find, ls, subagent |

### 2.2 Contained Freedom Principle

Each agent has maximal freedom within its role. The pipeline structure is the container that imposes the necessary constraints. Over-constraining individual agents reduces output quality:

- A good scout task: "Find everything relevant to authentication"
- A poor scout task: "List the authentication middleware function names in auth.ts"
- A good knowledge builder task: "Analyze the implications of restructuring X"
- A poor knowledge builder task: "List the 3 patterns in X"

The separation of concern is strict:
- Scout produces **data** (first-order facts)
- Knowledge builder produces **insight** (second-order observations) — never proposals
- Approach proposer produces **options** — never evaluation
- Critic produces **evaluation** — never alternatives
- Documenter produces **artifact** — never new analysis
- Edit planner produces **instructions** — never implementation
- Worker produces **code** — never planning

If an agent needs to do two things, call two agents.

### 2.3 Cumulative Context

A single shared context file (`workflow.md`) accumulates reasoning across all agents in a workflow. Each agent:
1. **Reads** the current context (all prior agents' output)
2. **Works** in its isolated process
3. **Appends** its output to the context file

The file grows monotonically:
```
┌─────────────────────────────────────┐
│ ## Scout Output                     │  ← first, grounding facts
│   File list, types, data flows      │
├─────────────────────────────────────┤
│ ## Knowledge Builder Output         │  ← second-order observations
│   Patterns, tensions, constraints   │
├─────────────────────────────────────┤
│ ## Approach Proposer Output         │  ← options
│   3 approaches with tradeoffs       │
├─────────────────────────────────────┤
│ ## Critic Output                    │  ← evaluation
│   Scores, verdict                   │
├─────────────────────────────────────┤
│ ## Documenter Output                │  ← final artifact
│   Goal, incentives, target          │
└─────────────────────────────────────┘
```

### 2.4 Two-Phase Workflow

Architecture and implementation are decoupled by user review:

**Phase 1 — Architecture**: Scout → Knowledge Builder → Approach Proposer → Critic → [User Decides] → Documenter

**Phase 2 — Implementation**: Edit Planner → [User Reviews] → Worker → [User Reviews] → Documenter

The same `workflowId` is reused across phases. The context file persists on disk — architecture exploration can happen today, implementation a week later, all sharing the accumulated reasoning.

### 2.5 Implementation

StructurED is implemented as a plugin for the `pi` coding agent harness. Each agent is defined as a Markdown file with YAML frontmatter specifying name, description, tools, and system prompt. The orchestration tool (`subagent`) spawns an independent `pi` process per agent invocation with `--mode json` for structured output capture.

The context file lives at `.pi/workflows/{workflowId}.md` — project-scoped, not in `/tmp`, surviving reboots and sessions.

## 3. Attention-Theoretic Motivation

The cumulative context design is motivated by the transformer attention mechanism's known properties:

### 3.1 U-Shaped Attention

Liu et al. (2023) demonstrated that LLMs exhibit a "lost in the middle" phenomenon: information in the middle of long contexts is less reliably attended to than information at the beginning or end. The cumulative context file exploits this:

- **Top**: Scout findings (hard facts, ground truth) — always attended
- **Middle**: Intermediate reasoning (KB analysis, proposals, critique) — naturally backgrounded
- **Bottom**: Latest section + agent's task prompt — fresh, immediately relevant

Each agent sees a progressively longer context, but the *shape* stays optimal. The top provides grounding; the bottom provides the immediate task; the middle provides reasoning continuity.

### 3.2 Token-Continuation Reasoning

LLMs are token-continuation machines. To generate the correct next token, they need the right "story" preceding it. The cumulative context builds this story step by step, in logical order from top to bottom. Each section is written by a different agent with a different perspective, but the progression is:

```
Facts → Insight → Options → Evaluation → Decision → Plan → Code
```

This natural ordering makes correct continuations more likely as the chain extends, converging toward code edits.

### 3.3 Compute Scaling Through Decomposition

Smaller models have limited context windows and attention capacity. By decomposing a complex task into multiple small-context invocations, StructurED effectively scales the compute available to solve a problem without scaling the model size or context window of any single invocation. Each agent sees only the relevant slice of the total reasoning.

## 4. Comparison with PLANSEARCH

| Dimension | PLANSEARCH (Wang et al.) | StructurED (ours) |
|---|---|---|
| Diversity mechanism | Combinatorial subset sampling of observations | Structural agent isolation |
| Context sharing | Same context, same model | Separate processes, append-only artifact |
| Observation generation | Single model, multiple prompts | Different agents with different tools/capabilities |
| Scaling | O(n²) combinatorial | Linear in number of agents |
| Applicability | Competitive programming (small observation space) | Architecture (large codebase, open-ended) |
| Model requirement | Strong single model | Can distribute across model tiers |
| User interaction | None (fully automated) | Human-in-the-loop between phases |

PLANSEARCH needed combinatorial enumeration because the *same model in the same context* would converge to the same ideas. Agent isolation achieves diversity structurally: a scout and a knowledge builder genuinely see different things because they have different prompts, tools, and objectives.

## 5. Empirical Observations

In extended use on the Moonlift compiler project (a typed, jump-first compiled language embedded in LuaJIT), StructurED was applied to two significant refactoring tasks:

### Task 1: Phase-Split BackCmd (Approach C)

Split a 100-variant flat enum into two phase-enforced enums (declaration + body). The critique correctly identified Approach C as lowest-risk (2/5) and best philosophy fit (5/5), leading to a clean 325-line refactoring with zero test breakage.

### Task 2: Schema-Aligned Sub-Enums (Approach A)

Applied schema-aligned sub-enum grouping on top of the phase-split foundation. 21 sub-enums replacing 127 flat variants. The knowledge builder identified 12 non-obvious edge cases (BoolNot field shape, VecMask arity variance, address synthesis interleaving) that guided the approach design. 654 insertions, 378 deletions, zero test breakage.

In both cases, the knowledge builder's second-order observations were critical. Observations like "shape information is silently discarded in four decoder paths" and "PtrAdd is a synthetic command with no schema home" were not visible to the scout (which only reports facts) but emerged from cross-referencing scout findings with the specific problem intent.

## 6. Limitations and Future Work

- **Token cost**: The cumulative context grows with each agent. Long-running workflows with many steps may approach context limits.
- **Agent definition quality**: Results depend on well-crafted agent prompts. Poorly-defined agents produce poor reasoning.
- **Single-point orchestration**: The main agent orchestrates manually. Future work could formalize the orchestration protocol.
- **Evaluation**: A systematic benchmark against PLANSEARCH on code generation tasks would quantify the structural decomposition advantage.

## 7. Related Work

- **PLANSEARCH** (Wang et al., 2024): Search over natural language plans through combinatorial observation sampling. Direct inspiration and baseline.
- **Tree of Thoughts** (Yao et al., 2023): Search over reasoning paths with BFS/DFS. Shares the insight that intermediate reasoning structure improves results.
- **ReAct** (Yao et al., 2022): Interleaved reasoning and action. Agent isolation extends this to multi-agent reasoning.
- **Constitutional AI** (Bai et al., 2022): Multiple LLM invocations critique and revise outputs. Shares the separation-of-concern pattern.

## 8. Conclusion

StructurED demonstrates that structural decomposition through isolated agent processes produces diverse, high-quality reasoning for complex architecture tasks. By replacing combinatorial sampling with structural separation, it achieves diversity through design rather than enumeration. The cumulative context pattern maps naturally to transformer attention mechanisms, producing optimal context shapes at each reasoning step.

The approach is KISS by design — nine agents, one file, two phases — and has been validated on real compiler refactoring tasks with zero test regressions.
