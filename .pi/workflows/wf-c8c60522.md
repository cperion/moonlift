```lua
local function binding_table_for_island(scan, island)
    local entries, seen = {}, {}
    for _, id in ipairs(island.holes or {}) do
        local expr = scan.splice_map[id] or id
        if not seen[expr] then
            seen[expr] = true
            entries[#entries + 1] = "[" .. quote_lua_string(expr) .. "] = (" .. expr .. ")"
        end
    end
    if #entries == 0 then return "" end
    return "{" .. table.concat(entries, ", ") .. "}"
end

local function name_anonymous_island(kind, text, name_hint)
    if not name_hint or name_hint == "" then return text end
    if kind == "func" and text:match("^%s*func%s*%(") then
        return (text:gsub("^(%s*func)%s*", "%1 " .. name_hint, 1))
    end
    if kind == "region" and text:match("^%s*region%s*[%(%-%>]") then
        return (text:gsub("^(%s*region)%s*", "%1 " .. name_hint, 1))
    end
    if kind == "expr" and text:match("^%s*expr%s*%(") then
        return (text:gsub("^(%s*expr)%s*", "%1 " .. name_hint, 1))
    end
    return text
end

local function transform_mlua(src)

    local scan = Parse.scan_document(src)
    ...
    for _, island in ipairs(scan.islands) do
        out[#out + 1] = src:sub(cursor, island.start - 1)
        local island_src = src:sub(island.start, island.stop)

        end

    end

end
```

## Relationships

- `.mlua` execution path:
  1. `lalin.loadstring` / `lalin.mlua_run.loadstring`
  2. `transform_mlua(src)`
  3. `Parse.scan_document(src)` finds raw Lalin islands.
  4. Each island becomes `lalin.<api>{splice bindings}[[island source]]`.
  5. Lua executes transformed carrier chunk.
  6. `lalin.region` / `lalin.func` parse strings via `chain.lua`.
  7. Region values are registered in `default_session.T._lalin_host_region_frags`.
  8. Later `emit name(...)` resolves through `open_expand` session environment; `emit @{frag}(...)` resolves through splice binding.

- Current raw region island shape:
  - Pure `.mlua` complete region:
    ```lua

    entry start()
        jump ok(next = 0)
    end
    end
    ```

  - Pure `.mlua` bodyless region header:

    ```lua
    local y = 1
    ```

  - Current `host.lua` region header does not reconstruct header + body; it parses body alone and ignores parse issues.

## Observations

- Body-only region islands are not currently a scanner concept. `entry` is skipped as Lua, and `scan_document` only begins islands at `func`, `region`, `expr`, `struct`, `union`, `extern`.

- Current scanner cannot terminate a pure `.mlua` region header at newline. It tokenizes until matching `end` or EOF. Parser can detect a header at newline/EOF, but by then scanner has already swallowed following Lua code.

- `region(...` with no explicit name gets name injected from Lua assignment by `name_anonymous_island`. This is transform-time text rewriting, not parser-level Lua scope resolution.

- `infer_lua_assignment_name` is shallow textual backward scanning:
  - Supports simple/dotted LHS rightmost segment.
  - Rejects comparison operators before `=`.
  - Does not model Lua locals, shadowing, multiple assignment, function scopes, table indexing, or `_ENV`.

- `@{}` binding evaluation in `.mlua` is generated Lua at the island’s original location, e.g. `lalin.region{["T"] = (T)}[[...]]`, so Lua’s actual current scope applies to splice expressions today.

- Name splices are role-typed:
  - Whole-token `region @{name}(...` is supported.
  - Partial-token `region r_@{name}(...` is not supported, despite some library examples containing this pattern.

- EOF/end edge cases:
  - `tokenize_island` returns a token window at EOF even if depth never returned to zero.
  - Region parser can interpret EOF after signature as header, but if scanner included Lua tail text, parser reports “unexpected token after region island” or other parse errors.
  - `skip_lua_long_bracket` supports `[=[...]=]`; `skip_lua_comment` only recognizes `--[[...]]`, not long comments with equals.
  - Scanner skips Lua strings/comments before looking for island starts, so `region` in `[[...]]` or `"..."` is ignored.

- Editor/document analysis uses the same scanner. Any scanner changes for body-only/header resolution likely affect:
  - hosted island anchors,
  - semantic tokens,
  - folding ranges,
  - document parts,
  - binding/navigation facts.

- Existing tests cover:
  - anonymous/inferred island names (`tests/test_anonymous_island_names.lua`);
  - `.mlua` splice tables and region fragment emit splices (`tests/test_mlua_splice_shapes.lua`);
  - diagnostics for malformed region islands (`tests/test_mlua_diagnostics.lua`);
  - named emit resolution/recursive region detection (`tests/test_region_normal_form_recursive.lua`).
  They do **not** appear to cover `lalin.region[[sig]][[body]]` header behavior directly.

### Approach A: Header Reference Line (`scan` then body)
- **Core idea**: Add a pure `.mlua` body island form where a standalone Lua header expression line is followed by a Lalin `entry`/`block` body and transformed into a call on that header object.

- **Key changes**:
  - `lua/lalin/parse.lua`
    - Teach `scan_document` to recognize body-only region islands beginning at `entry`/`block`.
    - Recognize a preceding standalone Lua expression/name line as the header target.
    - Improve region header scanning so `region(...)` can terminate at the signature line when no immediate `entry`/`block` follows.
  - `lua/lalin/mlua_run.lua`
    - Transform:
      ```lua
      local scan = region(x: i32; done: cont())
      scan
      entry start()
          jump done()
      end
      end
      ```
      into roughly:
      ```lua
      local scan = lalin.region[[region scan(x: i32; done: cont())]]
      scan[[entry start()
          jump done()
      end
      end]]
      ```
      or `scan{bindings}[[...]]` when body splices exist.
  - `lua/lalin/host.lua`
    - Fix `region_header.__call` so it reconstructs/parses header + body correctly.
    - Stop ignoring parse issues.
    - Support `{...}` binding-table form for body splices.
  - Tests:
    - Header declaration followed by implementation.
    - Shadowed local headers.
    - Body splices.
    - Bad body diagnostics.
    - Existing complete `region(...) entry ... end end` compatibility.

- **Tradeoff**: Optimizes for true Lua lexical scope resolution because the generated code calls the actual Lua variable in scope; sacrifices some syntactic explicitness and introduces a potentially surprising standalone identifier line.

- **Risk**:
  - Scanner ambiguity around ordinary Lua code shaped like:
    ```lua
    foo
    entry(...)
    ```
  - Requires careful diagnostics when `foo` is not a `region_header`.
  - Requires a solid `host.region` header implementation, including splices.

- **Rough sketch**:
  - Add a new scanned island kind like `region_impl_body`.
  - Detect `entry`/`block` bodies and associate them with the immediately preceding standalone Lua header expression line.
  - Emit `<header_expr>{bindings}[[body_src]]` or `<header_expr>[[body_src]]`.
  - Fix `region_header.__call` to combine stored header declaration with body source and fail on parse issues.
  - Add editor-analysis support so the body island has correct anchors/folds/tokens.

### Approach B: Explicit Pure Syntax Marker (`region scan` implementation)
- **Core idea**: Introduce a new pure `.mlua` implementation form that names the current-scope header explicitly using Lalin-like syntax.

- **Key changes**:
  - New syntax:
    ```lua
    local scan = region(x: i32; done: cont())

    region scan
    entry start()
        jump done()
    end
    end
    ```
    transformed to:
    ```lua
    local scan = lalin.region[[region scan(x: i32; done: cont())]]

    scan[[entry start()
        jump done()
    end
    end]]
    ```
  - `lua/lalin/parse.lua`
    - Distinguish normal region declarations from implementation references:
      - `region name(...)` = full/header region declaration.
      - `region name` followed by newline + `entry`/`block` = implementation of current-scope header variable `name`.
    - Add diagnostics for `region scan` without body, unknown/invalid body, or malformed target expression.
  - `lua/lalin/mlua_run.lua`
    - Add transform support for implementation islands.
    - Body splices become header-call bindings.
  - `lua/lalin/host.lua`
    - Same required `region_header.__call` fixes as Approach A.
  - Docs/tests:
    - Document `region name` as a pure `.mlua` implementation marker, not a region declaration.
    - Tests for local shadowing:
      ```lua
      local scan = region(...)
      do
          local scan = region(...)
          region scan
          entry ...
          end
          end
      end
      ```

- **Tradeoff**: Optimizes for readability and scanner robustness by requiring an explicit implementation marker; sacrifices the ultra-minimal “just write the body” feel.

- **Risk**:
  - Slight grammar/diagnostic compatibility change: today `region scan` is simply malformed; after this it has meaning.
  - Need clear rules for whether `region M.scan` or only `region scan` is allowed.
  - Still depends on fixing host header application.

- **Rough sketch**:
  - Extend island scanner to recognize `region <lua-header-ref>` with no parameter list.
  - Parse `<lua-header-ref>` conservatively at first, likely simple identifier only.
  - Scan the following `entry`/`block` body using existing region depth logic.
  - Emit `<lua-header-ref>{bindings}[[body]]`.
  - Runtime Lua scope performs actual header resolution.

### Approach C: Immediate Header/Body Pairing
- **Core idea**: Treat `local h = region(...)` followed immediately by a Lalin body as one split region declaration, combining the signature and body at transform time.

- **Key changes**:
  - New syntax:
    ```lua
    local scan = region(x: i32; done: cont())
    entry start()
        jump done()
    end
    end
    ```
    transformed directly to:
    ```lua
    local scan = lalin.region[[region scan(x: i32; done: cont())
    entry start()
        jump done()
    end
    end]]
    ```
  - `lua/lalin/parse.lua`
    - Change region scanning to stop at a header signature only when no immediate `entry`/`block` body follows.
    - Add special handling for “split header + immediate body”.
  - `lua/lalin/mlua_run.lua`
    - Detect a header island followed by an adjacent body island.
    - Merge the header source and body source into one normal `lalin.region[[...]]` quote.
    - Use existing assignment-name inference to name anonymous headers.
  - `lua/lalin/host.lua`
    - Not required for this feature path, because the transform emits a complete region quote.
    - Still advisable to separately fix documented `lalin.region[[sig]][[body]]`.
  - Tests:
    - Immediate split region.
    - Header-only declaration still returns a header when followed by Lua code.
    - Complete existing region syntax still works.
    - Diagnostics when `entry` appears after a non-region header.

- **Tradeoff**: Optimizes for minimal user syntax and avoids depending on broken header-call behavior; sacrifices flexibility because the body must be adjacent to the header declaration.

- **Risk**:
  - Ambiguous with accidental Lua identifiers named `entry`/`block`.
  - Does not support “declare header here, implement later” unless paired immediately.
  - Static pairing is textual, not true Lua runtime scope resolution.

- **Rough sketch**:
  - Teach scanner to identify bodyless `region(...)` headers without swallowing later Lua.
  - If the next meaningful source segment is `entry`/`block`, scan it as the body of that header.
  - In transform, replace both regions of source with one `lalin.region[[header + body]]`.
  - Preserve existing full-region syntax unchanged.
  - Add diagnostics for body without preceding header.

### Comparison

- Pick **Approach A** if the main goal is true Lua current-scope behavior and flexible “declare now, implement later” support.
- Pick **Approach B** if you want current-scope resolution but with clearer, grep-friendly syntax and fewer scanner ambiguities.
- Pick **Approach C** if you want the smallest implementation surface and best compatibility with existing `lalin.region[[full source]]` behavior, accepting that implementations must be adjacent to their headers.
