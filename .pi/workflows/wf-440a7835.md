## Critical (must fix)
- `lua/lalin/code_mem_facts.lua:200,208,301-318` - Raw pointer params correctly get `MemExtentUnknown`, but any resolved object immediately becomes `MemBoundsInObject`, `MemNonTrapping`, and often `movable`. This makes unbounded raw pointer loads/stores backend-nontrapping and movable without a bounds/lease/view proof, violating the architecture and creating unsafe metadata.
- `lua/lalin/lower_to_back.lua:112-116` - `LowerStrategyKernel` and `LowerStrategyClosedForm` are silently emitted as ordinary Code fragments. This makes Kernel/Schedule/ClosedForm plans decorative, not executable, and is a hidden fallback without `LowerIssueFallback`/reject propagation.
- `lua/lalin/code_value_facts.lua:188-192` - `ClosedFormFact` stores `ValueExprValue(param.value)` as the “closed form” and says the exact expression awaits a trip-count fact. That is not a closed form. `code_schedule_plan.lua:51-57` then schedules it as `ScheduleClosedForm`, so false semantic facts can drive lowering once closed-form emission is implemented.

## Warnings (should fix)
- `lua/lalin/code_kernel_plan.lua:97-133` - Kernel planning marks every counted loop as `KernelPlanned` with only a Flow proof. It does not reject unknown alias/dependence, trapping accesses, volatile/atomic effects, or unknown calls; `call_effects` also attaches all module calls to every loop without scoping.
- `lua/lalin/code_schedule_plan.lua:63-68` - Schedule planning always emits `SchedulePlanned` for every planned kernel. Removing bounds contracts cannot produce an explicit schedule rejection/fallback, contrary to the success criteria.
- `lua/lalin/code_to_back.lua:298-300,366-377,468-474` - View lowering still depends on a Lua `ctx.view_defs` side table and emits no executable representation for `CodeInstViewMake`. Projections of view params/loaded descriptors will fail unless backed by an in-module `CodeInstViewMake`, so views are not truly first-class at Back lowering.

## Critical (must fix)
- `lua/lalin/code_value_facts.lua:41-47` - Closed-form arithmetic series is not semantically correct for zero-trip / negative-trip loops, and likely not for non-divisible positive steps. It uses raw `(stop - start) / step` with no `max(0, …)`/ceil trip-count handling, but `lower_to_back.lua:760-767` replaces the loop with this expression. `tests/test_lower_to_back_closed_form.lua:47` only checks `n >= 0`, so runtime correctness is not proven for valid `i32` inputs like `n < 0`.
- `lua/lalin/code_kernel_plan.lua:98-105` - Write-loop dependence legality is satisfied by the existence of any `MemNoLoopCarriedDependence` in the loop. It does not require every write/read or write/write pair to be proven safe, so one safe pair can mask another unknown dependence and still produce `KernelPlanned`/`SchedulePlanned`.
- `lower_to_back.lua:516-517` - Scalar kernel emission reorders memory within a block by emitting all bindings/loads before all store effects. Combined with `code_mem_facts.lua:431` treating same-object/same-index as safe, a source sequence like `p[i] = 1; x = p[i]` can be lowered to load-before-store and return the old value. Current scalar tests only cover load-before-store on disjoint objects.
- `lua/lalin/kernel_emit_support.lua:61-73,207-215` + `lua/lalin/lower_to_back.lua:574-597` - Vector scheduling claims `ValueExprSelect`, `ValueExprCmp`, `Div`, and affine forms are executable, but the vector emitter only implements vector add/sub/mul and otherwise falls back to scalar lowering/splatting. This is fake vector capability: e.g. `dst[i] = select(src[i] < 0, 0, src[i])` can receive `ScheduleVector` but won’t emit a real vector compare/select. The vector JIT test only proves `x + 1`.
