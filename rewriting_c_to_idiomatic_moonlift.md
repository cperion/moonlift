# Rewriting C Software into Idiomatic Moonlift

## A Practical Guide to Redesigning Existing Code as Products, Protocols, Regions, and Sealed Functions

**Status:** methodology guide  
**Audience:** C programmers, systems programmers, Moonlift users  
**Goal:** explain how to take ordinary C software and redesign it into native, idiomatic Moonlift systems.

---

## Core Thesis

Do not translate C into Moonlift line by line.

C code already contains the machine you want, but it usually hides that machine in conventions:

- status codes
- tag fields
- `switch` statements
- `goto cleanup`
- mutable state structs
- callbacks
- function pointers
- global variables
- macro-generated code
- loops with hidden state
- error codes returned through integers
- side-channel output parameters
- implicit ownership rules

A correct Moonlift rewrite extracts the real machine from those conventions and rewrites it as:

```text
Products    = data that exists together
Protocols   = choices consumed by control
Regions     = machines that consume products and protocols
Blocks      = named states with product payloads
Jumps       = typed transitions between states
Emits       = composition by protocol filling
Functions   = sealed product-to-product boundaries
Lua         = generator of repeated concrete machines
Externs     = C ABI boundary
```

The strongest rule:

> If C code returns a value only so the caller can branch on it, the Moonlift rewrite should be a region protocol, not a function returning a result.

Another way to say it:

> C encodes protocols as data.  
> Moonlift writes protocols as control.

## 1. The Rewrite Mindset

### 1.1 Do not port syntax

Bad rewrite strategy:

```text
C function -> Moonlift function
C loop     -> Moonlift loop-like block
C enum     -> Moonlift union
C return code -> Moonlift return code
C callback -> Moonlift function pointer
```

This preserves the wrong abstractions.

Good rewrite strategy:

```text
C struct fields           -> products
C return-code convention  -> protocol
C enum/tag dispatch       -> consumer region
C loop state              -> block parameters
C goto labels             -> blocks
C cleanup path            -> resource protocol / scope
C callback interface      -> continuation protocol
C vtable/polymorphism     -> Lua-generated concrete regions or explicit consumer region
C macro specialization    -> Lua factory
C exported API            -> sealed function
```

### 1.2 C is usually an encoded state machine

Most C systems code is already a state machine. It is just encoded with integers, labels, flags, and conventions.

Example C:

```c
int parse_number(Parser *p, double *out, int *next) {
    int i = p->pos;
    int sign = 1;

    if (i >= p->len) return ERR_EOF;

    if (p->buf[i] == '-') {
        sign = -1;
        i++;
    }

    if (i >= p->len || !isdigit(p->buf[i])) {
        return ERR_BAD_NUMBER;
    }

    double value = 0;
    while (i < p->len && isdigit(p->buf[i])) {
        value = value * 10 + (p->buf[i] - '0');
        i++;
    }

    *out = sign * value;
    *next = i;
    return OK;
}
```

The return code is not real output. It is a branch selector.

Idiomatic Moonlift shape:

```moonlift
region parse_number(
    p: ptr(u8),
    n: index,
    i: index;

    ok: cont(value: f64, next: index),
    eof: cont(pos: index),
    bad_number: cont(pos: index))
```

The successful value and failure positions become continuation payloads. The caller fills each exit.

### 1.3 The target is not “safe C”

Moonlift is not simply C with better syntax.

The goal is to expose what C hides:

```text
what data exists
what state the machine is in
what can happen next
where each outcome goes
which boundary is ABI-facing
which repetition is generated by Lua
```

## 3. Classification Rules

### 3.1 Structs are usually products

C:

```c
typedef struct Buffer {
    uint8_t *data;
    size_t len;
    size_t cap;
} Buffer;
```

Moonlift:

```moonlift
struct Buffer
    data: ptr(u8)
    len: index
    cap: index
end
```

A struct is a product when all fields coexist. This is the easiest part of the rewrite.

### 3.2 Enums are not automatically unions

C:

```c
typedef enum TokenKind {
    TOK_IDENT,
    TOK_NUMBER,
    TOK_STRING,
    TOK_EOF,
    TOK_ERROR
} TokenKind;
```

Do **not** rewrite this as a semantic union by default.

A tag is an encoded fact. The semantic design is the consumer protocol.

Stored product:

```moonlift
struct Token
    kind: u16
    start: index
    len: index
end
```

Consumer protocol:

```moonlift
region consume_token(
    toks: ptr(Token),
    i: index;

    ident: cont(start: index, len: index),
    number: cont(start: index, len: index),
    string: cont(start: index, len: index),
    eof: cont(),
    error: cont(pos: index, code: i32))
```

The tag exists as storage. The choice exists as control.

### 3.3 Return codes are usually protocols

C:

```c
int read_packet(Socket *s, Packet *out);
```

where return values mean:

```text
1  = got packet
0  = would block
-1 = closed
-2 = malformed
```

Moonlift:

```moonlift
region read_packet(
    s: ptr(Socket),
    out: ptr(Packet);

    got: cont(),
    would_block: cont(),
    closed: cont(),
    malformed: cont(code: i32))
```

The return code was dispatch. Rewrite it as a continuation set.

### 3.4 Out-parameters are continuation payloads

C:

```c
int lookup(Table *t, Key k, Value *out);
```

Moonlift:

```moonlift
region lookup(
    t: ptr(Table),
    key: Key;

    found: cont(value: Value),
    missing: cont())
```

If `out` is only meaningful on success, it belongs in the success continuation payload.

### 3.5 Goto labels become blocks

C:

```c
retry:
    if (!try_lock(&q->lock)) goto retry;
    if (q->empty) goto empty;
    job = pop(q);
    goto got;

empty:
    unlock(&q->lock);
    return EMPTY;

got:
    unlock(&q->lock);
    *out = job;
    return OK;
```

Moonlift:

```moonlift
region claim_job(
    q: ptr(Queue);

    got: cont(job: ptr(Job)),
    empty: cont())
entry retry()
    if try_lock(&q.lock) == 0 then jump retry() end
    if q.empty then jump empty_locked() end
    let job: ptr(Job) = pop(q)
    jump got_locked(job = job)
end

block empty_locked()
    unlock(&q.lock)
    jump empty()
end

block got_locked(job: ptr(Job))
    unlock(&q.lock)
    jump got(job = job)
end
end
```

Labels that represent meaningful states should be named blocks.

### 3.6 Loops become blocks with state products

C:

```c
for (size_t i = 0; i < n; i++) {
    acc += xs[i];
}
return acc;
```

Moonlift:

```moonlift
func sum(xs: ptr(i32), n: index): i32
    block loop(i: index = 0, acc: i32 = 0)
        if i >= n then return acc end
        jump loop(i = i + 1, acc = acc + xs[i])
    end
end
```

The loop variables become block parameters. The next iteration becomes a jump.

### 3.7 Switch statements are consumer regions

C:

```c
switch (node->kind) {
case NODE_INT:
    return type_int(node);
case NODE_NAME:
    return type_name(node);
case NODE_CALL:
    return type_call(node);
default:
    return error(...);
}
```

Moonlift:

```moonlift
region visit_node(
    ast: ptr(Ast),
    node: NodeRef;

    int_node: cont(n: ptr(IntNode)),
    name_node: cont(n: ptr(NameNode)),
    call_node: cont(n: ptr(CallNode)),
    invalid: cont(code: i32))
```

Then the typechecker emits the visitor:

```moonlift
emit visit_node(ast, node;
    int_node = tc_int,
    name_node = tc_name,
    call_node = tc_call,
    invalid = bad_node)
```

The `switch` is not just syntax. It is the dispatch consumer. Name it.

### 3.8 Callbacks become continuations

C:

```c
typedef struct ParserCallbacks {
    void (*on_value)(Value *v, void *ud);
    void (*on_error)(int pos, int code, void *ud);
} ParserCallbacks;

void parse(Parser *p, ParserCallbacks *cb, void *ud);
```

Moonlift:

```moonlift
region parse(
    p: ptr(Parser);

    value: cont(v: ptr(Value)),
    error: cont(pos: index, code: i32))
```

Callbacks are often manually encoded continuation protocols. Moonlift makes them direct.

### 3.9 Vtables become generated concrete machines or explicit consumer regions

C:

```c
typedef struct ShapeVTable {
    double (*area)(Shape *);
    void (*draw)(Shape *, Canvas *);
} ShapeVTable;
```

Ask what is really happening.

If the set of shapes is known at generation time, use Lua to generate concrete specialized functions/regions.

If the runtime receives encoded shape records, write a consumer region:

```moonlift
region draw_shape(
    scene: ptr(Scene),
    shape: ShapeRef,
    canvas: ptr(Canvas);

    done: cont(),
    invalid: cont(code: i32))
```

Then inside it, dispatch to concrete product handlers.

Do not preserve vtables unless dynamic foreign ABI interop requires it.

### 3.10 Macros become Lua factories

C:

```c
#define DEFINE_VEC(T, Name) ...
DEFINE_VEC(int32_t, VecI32)
DEFINE_VEC(double, VecF64)
```

Moonlift:

```lua
local function make_vec(name, T, ctype)
    local Vec = struct @{name}
        data: ptr(@{T})
        len: index
        cap: index
    end

    local push = region @{name .. "_push"}(
        v: ptr(@{Vec}),
        value: @{T};

        ok: cont(),
        oom: cont())
    ...
    end

    return { Vec = Vec, push = push }
end
```

C macros generate text. Lua generates typed Moonlift values.

### 3.11 `#ifdef` becomes Lua platform selection

C:

```c
#ifdef _WIN32
    CreateThread(...)
#else
    pthread_create(...)
#endif
```

Moonlift stack:

```lua
local ffi = require("ffi")

if ffi.os == "Windows" then
    return require("thread.windows").make()
else
    return require("thread.pthread").make()
end
```

Each backend generates the correct externs and Moonlift regions.

### 3.12 Global state becomes an explicit product

C:

```c
static Config g_config;
static Arena g_arena;
static Logger g_logger;
```

Moonlift:

```moonlift
struct Runtime
    config: ptr(Config)
    arena: ptr(Arena)
    logger: ptr(Logger)
end
```

Pass `ptr(Runtime)` to regions that need it.

Global state is hidden input. Make it product data.

## 4. The Main Rewrite Algorithm

Use this sequence for each C subsystem.

### Step 1 — Find the real boundaries

Classify each public C function:

```text
external API boundary
internal helper
state-machine transition
callback target
platform syscall wrapper
hot kernel
```

Only external API boundaries should remain functions by default.

Internal multi-outcome helpers usually become regions.

### Step 2 — Extract products

For every concrete memory shape, define a product:

```text
C struct
array descriptor
handle wrapper
context object
buffer
queue
table
node pool
arena
parser state
```

Moonlift products are structs, pointers, views, and block parameter lists.

### Step 3 — Extract protocols

For each function, ask:

```text
What does its return value mean?
Which out-parameters are valid only on some returns?
Which errno/status values are distinct paths?
Which callbacks can fire?
Which switch cases does the caller handle?
What does the caller do immediately after the call?
```

These become continuations.

### Step 4 — Replace result passing with continuation wiring

C style:

```c
int r = parse(p, &value);
if (r == OK) { ... }
else if (r == ERR) { ... }
```

Moonlift style:

```moonlift
emit parse(p;
    ok = got_value,
    err = bad_parse)
```

### Step 5 — Turn local labels/states into blocks

Every loop, retry path, cleanup path, and branch state becomes a block.

Block parameters are the live variables of that state.

### Step 6 — Compose regions

Where C calls helper functions and switches on their return codes, Moonlift emits helper regions and fills their protocols.

### Step 7 — Seal the outer API

After the internal graph is idiomatic, create functions at the boundary.

A sealed function translates the rich protocol into the external convention:

```moonlift
func parse_api(p: ptr(Parser), out: ptr(Value)): i32
    return region: i32
    entry start()
        emit parse_value(p; ok = ok, err = err, eof = eof)
    end

    block ok(v: Value)
        *out = v
        return 1
    end

    block eof()
        return 0
    end

    block err(pos: index, code: i32)
        return 0 - code
    end
    end
end
```

This lets C callers keep C conventions while Moonlift internals remain protocol-native.

## 5. Mapping Table: C Idiom to Moonlift Redesign

| C idiom | What it hides | Moonlift redesign |
|---|---|---|
| `int` status return | Control protocol | Region continuations |
| `bool` return | Two-way protocol | `yes/no`, `found/missing`, etc. continuations |
| `errno` | Out-of-band protocol payload | `error: cont(code)` |
| Out-param valid only on success | Continuation payload | `ok: cont(value)` |
| `enum kind` + `switch` | Dispatch consumer | Product tag + consumer region |
| Tagged union struct | Delayed dispatch | Encoded products + consumer region |
| Callback table | Continuation protocol | Region exits |
| Function pointer vtable | Runtime polymorphic dispatch | Lua specialization or explicit region dispatch |
| `goto cleanup` | Resource protocol | Blocks + resource scope |
| `goto retry` | State transition | Block self/forward jump |
| `for/while` locals | State product | Block parameters |
| Macros | Untyped generation | Lua typed factories |
| `#ifdef` | Platform selection | Lua backend selection |
| Global variables | Hidden product input | Explicit context struct |
| `void *userdata` | Erased product | Typed context pointer |
| `void *` polymorphism | Type erasure | Lua-generated concrete variants |
| `malloc/free` scattered | Hidden ownership protocol | Scope/arena/resource policy |
| `switch` fallthrough | Implicit control | Explicit case branches and jumps |
| Long function with flags | Hidden state machine | Region with named blocks |

## 6. Rewriting Error Handling

C error handling is usually encoded control.

### 6.1 Return code pattern

C:

```c
int open_db(const char *path, DB **out);
```

Potential results:

```text
opened
not_found
permission_denied
corrupt
oom
```

Moonlift:

```moonlift
region open_db(
    path: ptr(u8);

    opened: cont(db: ptr(DB)),
    not_found: cont(),
    permission_denied: cont(),
    corrupt: cont(pos: index),
    oom: cont())
```

### 6.2 Preserve C API at the edge

If you need a C-callable API:

```moonlift
func open_db_api(path: ptr(u8), out: ptr(ptr(DB))): i32
    return region: i32
    entry start()
        emit open_db(path;
            opened = opened,
            not_found = not_found,
            permission_denied = denied,
            corrupt = corrupt,
            oom = oom)
    end

    block opened(db: ptr(DB))
        *out = db
        return 0
    end

    block not_found()
        return 1
    end

    block denied()
        return 2
    end

    block corrupt(pos: index)
        return 3
    end

    block oom()
        return 4
    end
    end
end
```

Internally: protocols. Externally: stable C return code.

### 6.3 Do not create error unions

Avoid:

```moonlift
union DbOpenResult
    opened(ptr(DB))
    not_found()
    permission_denied()
    corrupt(index)
    oom()
end
```

This only boxes a future branch. Use the protocol.

## 7. Rewriting State Machines

C state machines often look like this:

```c
switch (p->state) {
case S_START: ...
case S_STRING: ...
case S_ESCAPE: ...
case S_DONE: ...
case S_ERROR: ...
}
```

Rewrite strategy:

1. Keep persistent state fields only if the machine truly suspends between calls.
2. If the machine runs to completion in one call, state becomes blocks.
3. If it can suspend, the suspension/resume protocol becomes explicit.

### 7.1 Non-suspending parser

C:

```c
while (i < n) {
    switch (state) {
    case START: ...
    case ESCAPE: ...
    }
}
```

Moonlift:

```moonlift
region parse_string(
    p: ptr(u8),
    n: index,
    i: index,
    out: ptr(u8);

    done: cont(next: index, len: index),
    err: cont(pos: index, code: i32))
entry start(pos: index = i, len: index = 0)
    if pos >= n then jump err(pos = pos, code = 1) end
    let c: u8 = p[pos]
    if c == 34 then jump done(next = pos + 1, len = len) end
    if c == 92 then jump escape(pos = pos + 1, len = len) end
    out[len] = c
    jump start(pos = pos + 1, len = len + 1)
end

block escape(pos: index, len: index)
    if pos >= n then jump err(pos = pos, code = 2) end
    ...
    jump start(pos = pos + 1, len = len + 1)
end
end
```

### 7.2 Suspending parser

If the parser can stop and resume because it needs more input, that is a protocol:

```moonlift
region parse_step(
    st: ptr(ParserState),
    p: ptr(u8),
    n: index;

    done: cont(next: index),
    need_more: cont(saved_state: ptr(ParserState)),
    err: cont(pos: index, code: i32))
```

Do not hide suspension as `return PARSE_NEED_MORE`.

Name it.

## 8. Rewriting Data Structures

### 8.1 Keep layout explicit

C structs map naturally to Moonlift structs.

```c
typedef struct HashTable {
    Entry *entries;
    size_t cap;
    size_t len;
} HashTable;
```

Moonlift:

```moonlift
struct HashTable
    entries: ptr(Entry)
    cap: index
    len: index
end
```

### 8.2 Separate storage encoding from semantic dispatch

C:

```c
typedef struct Node {
    int kind;
    union {
        IntNode int_node;
        CallNode call_node;
    } as;
} Node;
```

Moonlift design:

```moonlift
struct NodeRef
    kind: u8
    index: u32
end

struct IntNode
    value: i64
end

struct CallNode
    fn: NodeRef
    args: ptr(NodeRef)
    argc: index
end

struct NodeStore
    ints: ptr(IntNode)
    calls: ptr(CallNode)
end

region visit_node(
    store: ptr(NodeStore),
    node: NodeRef;

    int_node: cont(n: ptr(IntNode)),
    call_node: cont(n: ptr(CallNode)),
    invalid: cont(code: i32))
```

This gives better memory layout too: dense arrays per concrete node type, smaller references, better locality.

### 8.3 Use handles instead of polymorphic blobs

C often uses:

```c
void *ptr;
int kind;
```

Moonlift should use typed references or handles:

```moonlift
struct ResourceRef
    kind: u16
    index: u32
end
```

Then a region consumes the handle:

```moonlift
region use_resource(
    rt: ptr(Runtime),
    r: ResourceRef;

    file: cont(fd: i32),
    socket: cont(fd: i32),
    timer: cont(t: ptr(Timer)),
    invalid: cont(code: i32))
```

Again: product storage, protocol consumption.

## 9. Rewriting APIs

### 9.1 Internal API should be regions

Any internal C API like this:

```c
int foo(Context *ctx, A *a, B *out);
```

should be questioned.

If the return code is meaningful, use a region:

```moonlift
region foo(
    ctx: ptr(Context),
    a: ptr(A);

    ok: cont(b: B),
    invalid: cont(code: i32),
    oom: cont())
```

### 9.2 External API should be functions

When exporting to C or Lua, seal the protocol.

```moonlift
func foo_api(ctx: ptr(Context), a: ptr(A), out: ptr(B)): i32
```

It may internally emit `foo`.

### 9.3 Do not expose internal protocols if the consumer is C

C cannot consume Moonlift continuation protocols directly. Keep the boundary conventional.

The API layer translates:

```text
Moonlift protocol -> C return code/out params
```

The internal system remains idiomatic.

## 10. Rewriting Callbacks and Event Systems

### 10.1 Callback tables are protocols

C:

```c
typedef struct Walker {
    void (*enter)(Node *, void *);
    void (*leave)(Node *, void *);
    void (*error)(int, void *);
    void *userdata;
} Walker;
```

Moonlift:

```moonlift
region walk_tree(
    tree: ptr(Tree),
    root: NodeRef;

    enter: cont(node: NodeRef),
    leave: cont(node: NodeRef),
    error: cont(code: i32),
    done: cont())
```

If `enter` and `leave` are repeated events, you may instead produce fact products into a sink. But the semantic choices are still protocols at the consuming boundary.

### 10.2 Event loop callbacks

C/libuv-style APIs may remain at the LuaJIT FFI layer. Moonlift should own hot machines.

Pattern:

```text
Lua/libuv receives event
Lua packs or points to product data
Moonlift region consumes event product and exits through protocol
Lua or Moonlift fills continuation behavior
```

## 11. Rewriting Memory Management

C code often mixes allocation policy with algorithmic logic.

Moonlift redesign should separate:

```text
ownership policy
borrowed computation
resource cleanup
hot machine
```

### 11.1 Identify ownership

For each pointer:

```text
Who allocated it?
Who frees it?
Can it escape?
Can it be stored?
Is it borrowed?
Is it mutable?
Does it alias?
```

### 11.2 Prefer host/arena ownership outside hot regions

A common Moonlift pattern:

```text
LuaJIT scope/arena/resource layer owns memory
Moonlift receives ptr/view
Moonlift computes
Moonlift does not allocate unless explicitly calling an allocator extern
```

### 11.3 Allocation as protocol

C:

```c
void *p = malloc(n);
if (!p) return ERR_OOM;
```

Moonlift internal:

```moonlift
region alloc_bytes(
    arena: ptr(Arena),
    n: index;

    ok: cont(p: ptr(u8), n: index),
    oom: cont())
```

If using LuaJIT-owned memory, the allocation may occur in Lua, and Moonlift receives the borrowed pointer.

### 11.4 Cleanup paths become blocks or scope defers

C:

```c
buf = malloc(...);
if (!buf) return ERR;
file = open(...);
if (!file) goto cleanup_buf;
...
cleanup_file:
    close(file);
cleanup_buf:
    free(buf);
```

Moonlift redesign options:

1. Move resource ownership to Lua scope/resource helper.
2. If inside Moonlift, name cleanup states as blocks.
3. If exported to C, seal cleanup convention in a function.

## 12. Rewriting Polymorphism

### 12.1 C function pointers

C:

```c
typedef int (*Hasher)(const void *key);
```

If the hasher is selected at generation time, use Lua specialization.

```lua
local function make_table(name, Key, hash_region)
    ...
end
```

If the hasher is truly runtime-provided through C ABI, keep it as a function pointer. But isolate the boundary.

### 12.2 Vtables

Vtables are runtime dispatch tables. Ask:

```text
Is this truly open-world dynamic dispatch?
Or is the set of cases known?
```

Known set: generate concrete machines with Lua.

Open-world ABI: keep function pointers, but wrap them at a sealed boundary.

Moonlift-native internal code should not become vtable-oriented unless it is intentionally modeling an ABI.

### 12.3 Object systems

C object-style code:

```c
obj->vtable->update(obj, dt);
```

Moonlift redesign:

```moonlift
region update_object(
    world: ptr(World),
    obj: ObjectRef,
    dt: f32;

    updated: cont(),
    removed: cont(),
    invalid: cont(code: i32))
```

The object reference is product data. The dispatch is protocol-consuming control.

## 13. Rewriting Macro Systems

C macros do three jobs:

```text
constants
specialization
boilerplate generation
```

Moonlift/Lua handles these better.

### 13.1 Constants

Use Lua to compute and splice:

```lua
local MAX_INLINE = 32
```

```moonlift
if len <= @{MAX_INLINE} then ...
```

### 13.2 Repeated fields

```lua
local fields = {}
for i = 0, N - 1 do
    fields[#fields + 1] = moon.field("lane" .. i, moon.f32)
end
```

```moonlift
local VecN = struct
    @{fields...}
end
```

### 13.3 Generated regions

```lua
local function expect_byte(name, byte, code)
    return region @{name}(p: ptr(u8), n: index, i: index;
        ok: cont(next: index),
        err: cont(pos: index, code: i32))
    entry start()
        if i >= n then jump err(pos = i, code = @{code}) end
        if p[i] == @{byte} then jump ok(next = i + 1) end
        jump err(pos = i, code = @{code})
    end
    end
end
```

Lua generation is typed and structured. Avoid string-concatenating whole functions.

## 14. Rewriting Platform Code

C platform code often uses:

```c
#ifdef _WIN32
...
#else
...
#endif
```

Moonlift stack:

```lua
local ffi = require("ffi")

local backend
if ffi.os == "Windows" then
    backend = require("platform.windows")
else
    backend = require("platform.posix")
end

return backend.make()
```

Each backend returns:

```text
extern declarations
product layouts
regions
sealed functions
Lua helpers
```

Platform differences are resolved before Moonlift object code is produced. Moonlift receives a concrete monomorphic backend.

## 15. Detailed Example: Rewriting a C Lexer

### 15.1 Original C shape

```c
typedef enum {
    TOK_IDENT,
    TOK_NUMBER,
    TOK_STRING,
    TOK_EOF,
    TOK_ERROR
} TokenKind;

typedef struct {
    TokenKind kind;
    size_t start;
    size_t len;
    int error;
} Token;

int next_token(const uint8_t *p, size_t n, size_t i, Token *out);
```

This function returns a status and writes a token. The token contains a kind, and later code switches on it.

### 15.2 Product extraction

Token storage is a product:

```moonlift
struct Token
    kind: u16
    start: index
    len: index
    code: i32
end
```

The tag is storage. It is not the semantic dispatch.

### 15.3 Lexing protocol

The lexer operation should expose what the caller needs immediately:

```moonlift
region next_token(
    p: ptr(u8),
    n: index,
    i: index;

    ident: cont(start: index, len: index, next: index),
    number: cont(start: index, len: index, next: index),
    string: cont(start: index, len: index, next: index),
    eof: cont(pos: index),
    error: cont(pos: index, code: i32))
```

If the caller only wants to store tokens, provide a storage adapter:

```moonlift
region next_token_store(
    p: ptr(u8),
    n: index,
    i: index,
    out: ptr(Token);

    stored: cont(next: index),
    eof: cont(pos: index),
    error: cont(pos: index, code: i32))
entry start()
    emit next_token(p, n, i;
        ident = store_ident,
        number = store_number,
        string = store_string,
        eof = eof,
        error = error)
end

block store_ident(start: index, len: index, next: index)
    out.kind = 1
    out.start = start
    out.len = len
    out.code = 0
    jump stored(next = next)
end
...
end
```

The protocol remains primary. Token storage is an adapter.

### 15.4 Consumer protocol

Later parser code should not switch on `Token` directly everywhere. Create a consumer region:

```moonlift
region consume_token(
    t: ptr(Token);

    ident: cont(start: index, len: index),
    number: cont(start: index, len: index),
    string: cont(start: index, len: index),
    eof: cont(),
    error: cont(code: i32),
    invalid: cont(code: i32))
```

This centralizes tag interpretation.

## 16. Detailed Example: Rewriting a C Parser

### 16.1 Original C style

```c
int parse_expr(Parser *p, Expr **out) {
    Token t;
    int r = next_token(p, &t);
    if (r != OK) return r;

    switch (t.kind) {
    case TOK_NUMBER:
        return parse_number_expr(p, &t, out);
    case TOK_IDENT:
        return parse_name_expr(p, &t, out);
    default:
        return ERR_EXPECTED_EXPR;
    }
}
```

### 16.2 Moonlift redesign

```moonlift
region parse_expr(
    p: ptr(Parser);

    expr: cont(e: ExprRef),
    expected_expr: cont(pos: index),
    lex_error: cont(pos: index, code: i32))
entry start()
    emit next_token(p.buf, p.len, p.pos;
        ident = got_ident,
        number = got_number,
        string = got_string,
        eof = got_eof,
        error = bad_lex)
end

block got_number(start: index, len: index, next: index)
    emit make_number_expr(p, start, len, next;
        ok = expr,
        oom = oom)
end

block got_ident(start: index, len: index, next: index)
    emit make_name_expr(p, start, len, next;
        ok = expr,
        oom = oom)
end

block got_string(start: index, len: index, next: index)
    jump expected_expr(pos = start)
end

block got_eof(pos: index)
    jump expected_expr(pos = pos)
end

block bad_lex(pos: index, code: i32)
    jump lex_error(pos = pos, code = code)
end
end
```

No `ParseResult`. No token union. No status-code cascade. Control is wired directly.

## 17. Detailed Example: Rewriting a C Queue

### 17.1 C style

```c
int queue_pop(Queue *q, Job **out) {
    if (q->closed) return QUEUE_CLOSED;
    if (q->head == NULL) return QUEUE_EMPTY;
    *out = q->head;
    q->head = q->head->next;
    return QUEUE_OK;
}
```

### 17.2 Moonlift region

```moonlift
region queue_pop(
    q: ptr(Queue);

    got: cont(job: ptr(Job)),
    empty: cont(),
    closed: cont())
entry start()
    if q.closed then jump closed() end
    if q.head == nil then jump empty() end

    let job: ptr(Job) = q.head
    q.head = job.next
    jump got(job = job)
end
end
```

### 17.3 Sealed function for C callers

```moonlift
func queue_pop_api(q: ptr(Queue), out: ptr(ptr(Job))): i32
    return region: i32
    entry start()
        emit queue_pop(q; got = got, empty = empty, closed = closed)
    end

    block got(job: ptr(Job))
        *out = job
        return 0
    end

    block empty()
        return 1
    end

    block closed()
        return 2
    end
    end
end
```

Internal code uses the protocol. C callers get the conventional API.

## 18. Detailed Example: Rewriting C Cleanup Code

### 18.1 C style

```c
int load_file(const char *path, Buffer *out) {
    FILE *f = fopen(path, "rb");
    if (!f) return ERR_OPEN;

    uint8_t *buf = malloc(MAX);
    if (!buf) {
        fclose(f);
        return ERR_OOM;
    }

    int r = fread(buf, 1, MAX, f);
    if (r < 0) {
        free(buf);
        fclose(f);
        return ERR_READ;
    }

    out->data = buf;
    out->len = r;
    fclose(f);
    return OK;
}
```

### 18.2 Moonlift redesign

Do not blindly translate cleanup. Decide ownership.

If LuaJIT or a SAR-style library owns resources, write:

```text
Lua opens/allocates/resources in scope
Moonlift fills borrowed buffer
Lua closes/frees through scope/resource discipline
```

If Moonlift owns the whole operation, encode outcomes:

```moonlift
region load_file(
    path: ptr(u8),
    out: ptr(Buffer);

    loaded: cont(),
    open_error: cont(code: i32),
    oom: cont(),
    read_error: cont(code: i32))
```

Internal cleanup states can be blocks:

```moonlift
block read_failed(buf: ptr(u8), f: ptr(FILE), code: i32)
    free(buf)
    fclose(f)
    jump read_error(code = code)
end
```

Cleanup is state transition, not magic.

## 19. When Not to Rewrite

Not every C thing must become Moonlift.

Keep C ABI or FFI when:

```text
the code is stable and external
the library is already battle-tested
the operation is OS-specific boilerplate
the boundary is not performance-critical
the function is simpler than the wrapper
the ABI must remain exactly C
```

Use Moonlift when:

```text
control flow is complex
branching is hot
status codes are everywhere
callbacks obscure the machine
macros generate lots of variants
parsers/codecs/protocols dominate
data layout matters
you want object/shared emission
you want Lua generation
```

The goal is not to rewrite the world. The goal is to rewrite the machine-shaped parts correctly.

## 20. Migration Strategy

### 20.1 Wrap first, redesign second

Start by exposing existing C functions through LuaJIT FFI or Moonlift externs. This gives you tests and comparison.

### 20.2 Choose one hot subsystem

Good candidates:

```text
parser
lexer
codec
queue
scheduler
hash table
protocol state machine
bytecode interpreter
validation pipeline
```

### 20.3 Write a behavior oracle

Use the old C implementation as the oracle.

Generate inputs. Compare outputs.

### 20.4 Redesign products and protocols

Do not look at C line by line. Write the product/protocol design.

### 20.5 Implement regions

Build internal regions first.

### 20.6 Seal the old API

Expose a function with the same C shape if needed.

### 20.7 Replace call sites gradually

C or Lua callers can keep using the sealed function while new Moonlift internals compose via regions.

## 21. C Rewrite Smell Catalog

When reading C, these are signals that a region/protocol is hiding.

### 21.1 Function returns small integer status

Almost always protocol.

### 21.2 Function has multiple out-parameters

Likely several continuation payloads.

### 21.3 Function returns `NULL`

Usually `found/missing`, `ok/oom`, or `opened/failed`.

### 21.4 Function writes `errno`

Out-of-band `error: cont(code)`.

### 21.5 Large switch on `kind`

Consumer region.

### 21.6 Struct contains `state`

Either persistent product state or a hidden block machine. Decide which.

### 21.7 Many booleans/flags

Often multiple protocols collapsed into product fields.

### 21.8 `goto fail`, `goto cleanup`, `goto retry`

Named blocks.

### 21.9 Callback table

Continuation protocol.

### 21.10 Macro family

Lua factory.

### 21.11 `void *userdata`

Hidden product context. Type it.

### 21.12 `#ifdef`

Lua platform generation.

## 22. Idiomatic Moonlift Rewrite Checklist

For each subsystem:

```text
[ ] I listed all C structs and mapped real products.
[ ] I identified all return-code meanings.
[ ] I replaced internal status returns with continuation protocols.
[ ] I converted out-parameters into continuation payloads where appropriate.
[ ] I replaced tag-switch semantics with consumer regions.
[ ] I kept tags only as storage encodings, not semantic design.
[ ] I converted loops into blocks with explicit state products.
[ ] I converted goto labels into named blocks.
[ ] I isolated C ABI compatibility in sealed functions.
[ ] I moved macro repetition into Lua factories.
[ ] I moved platform selection into Lua.
[ ] I made ownership and borrowing explicit.
[ ] I kept extern calls at real world boundaries.
[ ] I avoided semantic unions/result boxes.
[ ] I used emit for internal composition.
[ ] I used func only where the protocol is sealed.
```

## 23. The Final Rule

A correct C-to-Moonlift rewrite is not judged by whether the Moonlift code resembles the C code.

It is judged by whether the hidden machine has become explicit.

C asks the reader to remember conventions:

```text
-1 means error
0 means EOF
positive means bytes read
out is valid only on success
state 3 means inside escape
kind 7 means call node
callback may fire before return
```

Moonlift should make those conventions source-level structure:

```text
error: cont(code)
eof: cont()
got: cont(bytes)
escape: block(pos, len)
call_node: cont(ptr(CallNode))
value: cont(v)
```

The rewrite succeeds when:

```text
products contain only facts
protocols contain all choices
regions contain the machines
blocks contain the states
jumps contain the transitions
functions contain only sealed boundaries
Lua contains the generation policy
```

That is idiomatic Moonlift.

---

## 24. Summary

To rewrite C into Moonlift:

1. Do not port functions; recover machines.
2. Map C structs to products.
3. Treat return codes, enums, callbacks, and tag switches as protocols.
4. Treat loops and gotos as block/jump state machines.
5. Treat macros and platform branches as Lua generation.
6. Treat exported APIs as sealed functions.
7. Treat stored tags as encodings, not semantic unions.
8. Keep choices in continuation protocols.
9. Keep data as products.
10. Compose with `emit`, seal with `func`.

The shortest version:

> C encodes control as data and convention.  
> Moonlift writes control as typed protocols.  
> Rewrite by extracting the protocol, not by copying the function.
