# Lua JIT Harness

The JIT harness is the offline machinery that builds the Lua VM's copy-and-patch JIT stencil library.

**Design reference:** `LUA_STENCIL_HARNESS_DESIGN.md` (in parent directory)

## Overview

The harness transforms real Lua programs into an empirical stencil library via an 11-step pipeline:

1. **Corpus discovery** — Find and normalize Lua programs (AWFY test suite)
2. **Bytecode profiling** — Extract opcode patterns from real programs
3. **Fact collection** — Canonicalize runtime observations into facts
4. **L0 seed selection** — Build primitive opcodes + manually selected useful stencils
5. **Layer closure** — Generate L1-L4 candidates by bounded-arity composition
6. **Kernel emission** — Emit Moonlift source from candidate specifications
7. **Compilation** — Compile through Moonlift/Cranelift to object files
8. **Object mining** — Extract bytes, holes, relocations, clobbers
9. **Verification** — Verify candidates against semantic contracts
10. **Benchmarking** — Measure performance on corpus
11. **Selection & export** — Rank candidates, select winners, export runtime library

## L0 Seed Library

27 manually selected stencil families:

- **Value ops:** LOADI, LOADK, MOVE
- **Arithmetic:** ADD_i64 (known/guarded), SUB, MUL, DIV
- **Comparisons:** EQ, LT, LE, COMPARE_BRANCH
- **Control:** TEST_JMP, JMP, FORLOOP, FORPREP
- **Tables:** GETTABLE, SETTABLE, GETFIELD, SETFIELD
- **Calls:** CALL (generic and compound)
- **Returns:** RETURN1, RETURN (varargs)
- **Projections:** PROJECT_slots (1-3)

These are the seeds for L1 generation. All 27 marked as manually seeded, priority=100.

## L1 Generation

L1 is generated by cumulative unit closure over L0 with unit arity 1..4, then expanded by declared fact axes. Profile data is a slow/unobserved filter and scheduler; it does not replace the closure semantics.

A candidate identity is:

```text
PatternWindow × FactSet × ShapeKind × Lowering
```

Examples:

```text
GETFIELD @ GETFIELD:raw_string_slot
MOVE|MOVE @ MOVE:move_def;MOVE:move_uses_previous_def
ADDI|RETURN1 @ ADDI:i64;RETURN1:returns_previous_def
```

Every candidate carries shape/legalization metadata:

```text
shape_kind          fallthrough | pure_rewrite | guarded_pure_rewrite |
                    branch_or_control_boundary | call_boundary |
                    effect_boundary | terminal_return
lowering            generic_opcode_sequence | move_move_forward | ...
continuation        fallthrough_pc_plus_arity | fallthrough_pc_plus_2 |
                    branch_pc_or_side_exit | return_boundary | ...
legalization_source default_opcode_shape | operand_fact_rewrite_schema | ...
```

The `L1_rewrite*` outputs are retained as a view/report over rewrite-shaped candidates. They are not a later peephole optimizer pass; they show candidates whose facts legalize a non-naive lowering.

## Benchmarking and Selection

Layer generation is discovery, not final admission into the runtime library. Before a layer can seed L2 or runtime export it must pass selection:

```bash
luajit tools/jit_harness/harness.lua select-layer \
  --layer build/.../L1/layer.json \
  --profile build/.../profile/static_profile.json \
  --out build/.../selection_L1 \
  --max-bench 512 \
  --max-selected 128
```

Selection performs:

1. shape/legalization verification,
2. hot-candidate preselection from profile frequency,
3. lowering-plan construction from facts,
4. kernel emission,
5. object compilation,
6. profitability benchmarking,
7. winner selection by score/quota.

The current benchmark mode is `profitability_model_v1`: it is deterministic and VM-shaped, using opcode baseline cost, observed frequency, candidate shape/lowering, concrete lowering support, side-exit risk, and object size. Boundary-only stencils that merely side-exit to the interpreter are legal but not selected as profitable.

Facts are consumed through `lowering_plan.lua`. The GCC backend now has a lowering for every fact emitted by the current L0/L1 fact vocabulary. Concrete guarded lowerings cover generic/raw table reads, generic/raw table writes, GETUPVAL/GETTABUP, integer arithmetic, numeric DIV, TEST/JMP, i64 FORPREP/FORLOOP, variable/fixed returns, known/generic call-boundary guards, local rewrites, and branch lowerings for `EQ`, `EQK`, `LT`, and `LE`. `NEWTABLE:generic` is represented as an allocator boundary because the experimental VM itself has no allocator-backed newtable implementation yet.

Artifacts:

```text
selection_L1/
  bench.json              # per-candidate profitability data
  compile.json            # object compile results
  selection.json          # winner/rejection decisions
  selected_layer.json     # original layer plus selected[]
  selection_summary.md    # human report
```

Only `selected_layer.json:selected` should seed L2. Do not compose all generated L1 candidates.

### Cumulative layer closure

Layer closure is cumulative. `--max-arity 4` means four composed **units**, not four underlying opcodes.

```text
L0 = primitive opcode/fact/rewrite units
L1 = all valid arity 1..4 combinations from {L0}
L2 = all valid arity 1..4 combinations from {L0, selected L1}
L3 = all valid arity 1..4 combinations from {L0, selected L1, selected L2}
LN = all valid arity 1..4 combinations from cumulative selected lower layers
```

The profile is used as a slow/unobserved filter and scheduler, not as the definition of the candidate universe. The generator walks the cumulative atom product, rejects invalid shape composition, applies native artifact budgets, and prunes unobserved/slow profile spans.

For L2+, lower-layer atoms are already compiled away. Their opcode lists remain selector/profile metadata only; native composition uses compiled artifacts (`native_artifact`/`compiled_object`) and budgets such as bytes, holes, relocs, exits, and patch sites. `max_profile_span` may limit profile lookup over flattened opcode metadata, but opcode count is not the semantic codegen budget for native-composed layers.

Example L2:

```bash
luajit tools/jit_harness/harness.lua build-layer \
  --layer 2 \
  --seed-manifest build/.../l0_seed_manifest_fact.json \
  --from-layers build/.../selection_L1/selected_layer.json \
  --profile build/.../profile/static_profile.json \
  --out build/.../L2 \
  --max-arity 4 \
  --max-opcodes 16 \
  --max-size 1200
```

Example L3:

```bash
luajit tools/jit_harness/harness.lua build-layer \
  --layer 3 \
  --seed-manifest build/.../l0_seed_manifest_fact.json \
  --from-layers build/.../selection_L1/selected_layer.json,build/.../selection_L2/selected_layer.json \
  --profile build/.../profile/static_profile.json \
  --out build/.../L3 \
  --max-arity 4 \
  --max-opcodes 32
```

Higher layers inherit fact-specialized shapes from selected lower-layer atoms. They do not replace layer closure with motif tiling.

### GCC stencil backend

Selection can emit C stencils and compile them with GCC instead of Moonlift:

```bash
luajit tools/jit_harness/harness.lua select-layer \
  --layer build/.../L2_motif/layer.json \
  --profile build/.../profile/static_profile.json \
  --out build/.../selection_L2_gcc \
  --backend gcc \
  --max-bench 124 \
  --max-selected 64
```

The GCC backend emits one C function per stencil with the same `jit_stencil_v0` ABI shape and compiles using `gcc -O3 -c` plus freestanding/no-unwind flags. This path is intended to mine optimized native stencil shapes from a mature C optimizer while keeping the same candidate/fact/shape selection pipeline.

## Pipeline Architecture

```
        AWFY Corpus
            ↓
        (Normalize files)
            ↓
        Bytecode patterns
            ↓
        (Profile & extract facts)
            ↓
        L0 Seed Manifest
            ↓
        (Generate L1..L4 by closure)
            ↓
        L1-L4 Candidates
            ↓
        (Emit Moonlift kernels)
            ↓
        (Compile → object files)
            ↓
        (Mine bytes/holes/relocs)
            ↓
        (Verify semantics)
            ↓
        (Benchmark on corpus)
            ↓
        (Score & rank)
            ↓
        Runtime Library
            ├── stencil_library.h (C)
            ├── stencil_manifest.json (metadata)
            └── binary blobs (machine code)
```

## Design Principles

1. **Offline only** — Harness never runs in hot runtime path
2. **Evidence-driven** — Only stencils observed in real programs
3. **Bounded-arity closure** — L1 pairs → L2 from L0+L1 → L3 from L0+L1+L2, etc.
4. **Measurement** — Generated → verified → benchmarked → ranked → pruned
5. **Empirical** — Real AWFY corpus is ground truth, not speculation
6. **No hand-written families** — All stencils from algorithm + evidence, not manual designs
